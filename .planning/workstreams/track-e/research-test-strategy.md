# Research: Discord & AI Bot Testing Strategy

## Overview
This document outlines best practices and patterns for testing AI-powered Discord bots using local models (Ollama). The focus is on moving beyond unit tests to reliable scenario-based integration testing.

---

## 1. Discord Testing Patterns
Testing `discord.py` bots requires simulating the Discord API without connecting to live servers.

### Mocking Discord Objects
- **Spec-based Mocking**: Use `unittest.mock.MagicMock(spec=discord.Message)` to ensure mocks only have attributes/methods present in the real class.
- **AsyncMock**: For all coroutines (`send`, `add_reaction`, `edit`), use `unittest.mock.AsyncMock`.
- **Key Objects to Mock**:
    - `discord.Message`: Set `content`, `author`, `channel`, and `guild`.
    - `discord.Context`: The primary container for command execution. Mock `send`, `reply`, and `author`.
    - `discord.Interaction`: Crucial for slash commands. Mock `response.send_message` and `followup.send`.

### Specialized Libraries
- **dpytest**: A popular extension specifically for `discord.py` that provides a fake state and allows "sending" messages to a mock bot.
- **accord.py**: A newer testing library that simplifies the setup of mock Discord environments.
- **Disfake**: Useful for generating realistic fake Discord objects for testing purposes.

---

## 2. AI Model Testing Strategies (Ollama/OpenAI)
AI testing must handle non-determinism and the latency of local inference.

### Mocking Ollama/OpenAI Calls
- **Boundary Mocking**: Do not mock internal logic; mock the network boundary (e.g., `ollama.chat` or `openai.ChatCompletion.acreate`).
- **Response Stubs**: Use `AsyncMock` to return predefined JSON structures matching the Ollama/OpenAI API.
    ```python
    mock_ollama.chat.return_value = {
        'message': {'role': 'assistant', 'content': 'Hello! I am your DM.'},
        'done': True
    }
    ```

### Replay & Snapshot Testing
- **VCR.py / Cassettes**: Record real model responses once and "replay" them in subsequent test runs. This ensures speed and determinism while testing how the bot handles complex LLM output.
- **Snapshot Testing**: Use `pytest-snapshot` to verify that the prompt generated by the bot remains consistent, even if the model isn't called.

---

## 3. Scenario-Based Test Structure
Scenarios should reflect real user journeys (e.g., "Player joins, starts combat, rolls a hit").

### Flow Categories
- **Normal Flow**: The "Happy Path" (e.g., `!join` -> `!roll` -> Success message).
- **Error Flow**: Handling invalid input (e.g., `!roll 1d0`, `!cast` with no mana).
- **Recovery Flow**: Handling AI failures or timeouts (e.g., Model returns garbage, connection to Ollama drops).

### Test Isolation
- **Per-Test Database**: Use SQLAlchemy with a unique `sqlite:///:memory:` or a temporary PostgreSQL schema for every test to ensure state doesn't leak.
- **Context Management**: Use `pytest.fixture` to reset the bot state and clear any cached data before each scenario.

---

## 4. Recommended Frameworks & Tools
Beyond standard `pytest`, these tools enhance AI/Discord testing:

- **pytest-asyncio**: Mandatory for `discord.py` and async AI calls.
- **pytest-bdd**: Allows writing tests in Gherkin (`Given/When/Then`). Excellent for "Dungeon Master" scenarios where non-technical stakeholders might define the flow.
- **Hypothesis**: Use for "Property-Based Testing" to feed random (but valid) dice strings or player names to the bot to find edge-case crashes.
- **DeepDiff**: Useful for comparing large state snapshots (e.g., checking if the entire "Combat State" changed correctly after a turn).

---

## 5. Actionable Recommendations

1.  **Implement a `FakeInteraction` Factory**: Create a utility that generates a fully-formed `discord.Interaction` or `discord.Context` with nested mocks (Guild, Member, Channel) to reduce boilerplate in tests.
2.  **Standardize LLM Mocks**: Create a "Model Mocking" fixture that can switch between "Instant Success", "Delayed Response", and "API Error" to test bot resilience.
3.  **Use Gherkin for Combat Scenarios**: Define complex rules (like multi-attack or spell saving throws) in `.feature` files using `pytest-bdd`. This makes the "Rules Engine" verifiable by reading plain English.
4.  **Prompt Regression Tests**: Add tests that specifically assert the *content of the prompt* being sent to Ollama, ensuring that changes to the code don't accidentally degrade the AI's "Instructions".
5.  **Integration "Smoke" Tests**: Maintain a small set of tests that *do* call a live (but small/fast) local model to ensure the integration hasn't broken due to API updates.
